Cursor·AI를 PM이 어떻게 썼나

코드를 처음부터 한 줄씩 짜기보다, 의도·화면·에러 메시지를 말로 넘기고 AI가 초안을 만들게 했습니다.

  • 기획 → 코드 — “이런 흐름의 토론 UI”, “맛집일 때만 카카오 쓰고 싶다”처럼 요구를 문장으로 던짐
  • 반복 루프 — 화면을 써 보고 → 톤이 이상하면 → 프롬프트·금지어·역할을 같이 고침 (스펙만으로는 안 보이던 디테일이 여기서 드러남)
  • 막힐 때 — TypeScript·ESLint·빌드 에러 로그를 붙여 넣고 수정 요청, npm run build까지 맡겨서 배포 가능 상태로 맞춤
  • 배포 — GitHub·Vercel 연결, 환경 변수 위치(.env.local vs 대시보드)도 단계별로 질문하며 정리

무엇을 만들었나 · 사용자에게 주는 가치

한 줄 고민을 던지면, 성향이 다른 전문가 3인이 토론하고, 합의·투표형 결론·후속 선택지(2차전)까지 줍니다.

  • 입력 — 맥북 살까, 와인 마실까, ○○ 맛집 같은 일상 의사결정 한 줄
  • 출력 — 4턴 토론 타임라인, MC 정리, 지난 토론 저장, 마음에 드는 턴·합의 북마크
  • 기획 포인트 — 정답 검색기가 아니라 관점을 부딪쳐 결정을 앞당기는 놀이터
  • PM 역할 — 룰·톤·분기 조건을 정하고, AI는 그걸 코드·프롬프트로 옮기는 조수

구현 ① · 뼈대

기술 구조 — 어디서 무엇이 돌아가나

프론트는 Next.js(App Router) + React, 토론 생성은 서버 API 한 곳으로 모았습니다.

  • 엔드포인트 — POST /api/debate가 질문·후속 선택을 받고 JSON으로 토론 묶음을 돌려줌
  • AI 호출 — 기본은 OpenAI로 멀티턴 토론 생성, 실패·조건에 따라 로컬 템플릿 폴백
  • 맛집만 카카오 — 질문에 「맛집」이 있을 때만 카카오 키워드 검색 → 실제 업장 목록을 컨텍스트로 넣음
  • 분기 — 전자기기·음주 의사결정 등은 키워드로 감지해 전용 페르소나·톤으로 분리

요청 흐름: 브라우저 → /api/debate → OpenAI 또는 로컬 팩 · 맛집이면 카카오 검색 결과를 끼워 넣음 → JSON을 카드 UI로 표시.

구현 ② · 연동·품질·배포

카카오 · 톤 잡기 · Vercel

  • 맛집만 카카오 — 질문에 「맛집」 있을 때만 호출 → 나머지는 키 없이도 동작
  • 콘솔 — 카카오 앱에서 Open Map & Local 활성화(꺼지면 403)
  • 환경 변수 — 로컬은 .env.local · 배포는 Vercel → Settings → Environment Variables(GitHub에는 안 올라감)
  • Vercel — 저장소 연결 후 main push마다 빌드 · 프로덕션 URL로 데모 가능
  • 검색 안정화 — 질문에서 대화 꼬리 제거, 0건이면 사용자에게 시도한 검색어와 함께 안내
  • 답변 톤 — 초기엔 논문체·가짜 통계 느낌 → Cursor로 프롬프트를 여러 번 고쳐 구어체·금지 규칙·역할을 쌓으며 개선
  • 폴백 — OpenAI 실패 시 로컬 템플릿 · 맛집은 카카오 목록 기반
